Phase 5: Production (Sprint Loop)
PENDING ▾

O ciclo principal de produção. Cada sprint implementa stories do backlog até que o jogo esteja completo.

Estrutura de cada Sprint

📋
Sprint Plan
/sprint-plan new
🔍
Story Readiness
/story-readiness [story]
🛠️
Implement
/dev-story [story]
✅
Story Done
/story-done [story]
5.1 Story Readiness (per-story) /story-readiness
Antes de implementar cada story, roda /story-readiness para verificar se ela ainda está válida (ADRs não mudaram, GDD não foi alterado, etc).
5.2 Implementation /dev-story
Roteia automaticamente para o agente programador correto (gameplay, engine, ai, ui, tools). Implementa código + testes contra os acceptance criteria.
5.3 Story Completion /story-done
8-phase review: verifica cada AC, checa desvios de GDD/ADR, prompt para code review, gera relatório. Atualiza status da story.
5.4 Sprint Tracking /sprint-status
Snapshot rápido do progresso lendo sprint-status.yaml. 30 linhas.
5.5 Sprint Review /retrospective → /sprint-plan new
Ao final de cada sprint: análise de velocity, blockers, melhorias. Depois planeja o próximo sprint.
Other Production Tools
/content-audit — compara conteúdo planejado vs implementado
/scope-check — detecta scope creep
/propagate-design-change — quando um GDD muda, propaga para ADRs afetados
/team-combat, /team-ui, etc — orquestração multi-domínio
/milestone-review "alpha" — revisão de milestone

As 149 stories já existem em production/epics/[slug]/story-*.md. Cada uma tem:

  • TR-IDs referenciando requisitos técnicos
  • ADR references (só de Accepted ADRs)
  • Control manifest version date
  • Engine-specific implementation notes
  • Acceptance criteria do GDD
  • QA test cases (da Phase 4b do create-stories)

Ordem de implementação recomendada: Foundation → Core A → Core B → Core C → Feature → Presentation. Dentro de cada camada, seguir a ordem de dependência das stories.
